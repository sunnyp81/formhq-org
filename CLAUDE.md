- 2026-07-10 — Fixed a full-site outage: all 21 pages (including the homepage) were returning HTTP 404 on formhq.org and formhq-org.pages.dev. Root cause: `@astrojs/cloudflare` (installed by the Jul 7 commit's implicit lockfile, adapter itself present since Apr 3) auto-enables Cloudflare KV Sessions and an Images binding by default whenever `session.driver` isn't set, which forces Astro into server build mode even with `output: 'static'`, splitting the build into `dist/client` (static HTML) + `dist/server` (a worker entrypoint requiring ASSETS/SESSION/IMAGES bindings). This repo's CF Pages project is a classic git-integration static deploy with no such bindings configured, so the resulting worker/routing failed and every request 404'd. `wrangler pages deployment list` showed 3 straight build FAILUREs (f54f14b, ac7b77d, df2997a) before the Jul 7 wrangler.jsonc addition happened to fix the CI build (it needed a compatibility_date to read) without fixing the routing, so the "successful" Jul 7 deploy was live-404 the whole time despite the brain's "Verified live: 200" note (that check was likely run against a stale/cached response before the new deploy fully propagated). Confirmed via grep: zero usages of `Astro.locals.runtime`, sessions, or Cloudflare image transforms anywhere in `src/` — the adapter was pure dead weight for this fully static site. Fix: removed `adapter: cloudflare()` and the `@astrojs/cloudflare` import from `astro.config.mjs`, uninstalled the npm dependency (`npm uninstall @astrojs/cloudflare`, package.json + package-lock.json updated). Build now reports `mode: "static"` and produces a flat `dist/` with all 21 `index.html` files at the expected paths (matches CF Pages' configured output directory `dist`). Pushed as `b37020d`; CF Pages auto-built and deployed (`14d5fa68`, status Active).
